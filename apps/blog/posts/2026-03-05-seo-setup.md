# SEO Setup

아무리 글을 써도 검색에 안 나오면 의미가 없으니까 SEO를 진행해보자.  
Next.js App Router는 SEO에 필요한 대부분의 기능을 내장하고 있어서 설정이 생각보다 간단했다.

## metadata API

Next.js App Router는 `metadata` 객체를 export하면 자동으로 `<head>` 태그를 생성해준다.

### 루트 layout.tsx — 사이트 기본 metadata

```tsx
import type { Metadata } from "next";

export const metadata: Metadata = {
  metadataBase: new URL(process.env.NEXT_PUBLIC_BASE_URL!),
  title: {
    default: "Playground",
    template: "%s | Playground", // 하위 페이지 이름 템플릿: "Posts | Playground"
  },
  description:
    "실무에서 새로운 기능 실험까지! 새로운 도약을 위한 자유로운 실험실입니다.",
  openGraph: {
    title: "Playground",
    description:
      "실무에서 새로운 기능 실험까지! 새로운 도약을 위한 자유로운 실험실입니다.",
    url: process.env.NEXT_PUBLIC_BASE_URL,
    siteName: "Playground",
    locale: "ko_KR",
    type: "website",
  },
  twitter: {
    card: "summary_large_image",
    title: "Playground",
    description:
      "실무에서 새로운 기능 실험까지! 새로운 도약을 위한 자유로운 실험실입니다.",
  },
};
```

`template: "%s | Playground"`를 설정하면 각 페이지에서 `title: "Posts"`만 써도 브라우저 탭에는 `Posts | Playground`로 표시된다.

```tsx
// 각 페이지 예시
export const metadata: Metadata = {
  title: "Posts",
  description: "모든 글 목록입니다.",
};
```

`metadataBase`는 og 이미지나 URL 같은 상대 경로를 절대 경로로 변환할 때 기준이 되는 URL이다. 없으면 og 이미지 경로가 제대로 생성되지 않는다.

### 포스트 상세 페이지 — 동적 metadata

```tsx
export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { slug } = await params;
  const post = publishedPosts.find((p) => p.id === slug);

  if (!post) return {};

  return {
    title: post.title,
    description: post.description ?? post.title,
    openGraph: {
      title: post.title,
      description: post.description ?? post.title,
      url: `${BASE_URL}${post.url}`,
      type: "article",
      publishedTime: post.date,
      tags: post.tags,
    },
    twitter: {
      card: "summary_large_image",
      title: post.title,
      description: post.description ?? post.title,
    },
  };
}
```

`description`은 Velite 스키마에 optional 필드로 추가했다. 없으면 `post.title`을 fallback으로 사용한다. 나중에 `gen-description` 스크립트로 자동 생성할 예정이다.

## BASE_URL 중앙 관리

metadata, sitemap, JSON-LD 등 여러 곳에서 도메인 URL을 사용한다. 중복을 줄이기 위해 환경변수와 상수 파일로 관리했다.

```bash
# .env
NEXT_PUBLIC_BASE_URL=https://playground-two-lemon.vercel.app
```

```typescript
// src/shared/lib/constants.ts
export const BASE_URL =
  process.env.NEXT_PUBLIC_BASE_URL ?? "http://localhost:3000";
```

`NEXT_PUBLIC_` 접두사를 붙여야 클라이언트 사이드와 Edge Runtime에서도 접근할 수 있다. Vercel 대시보드 환경변수에도 동일하게 등록해야 배포 환경에서 정상 동작하기 때문에 추가했다.

## og 이미지

카카오톡이나 슬랙에서 링크를 공유했을 때 썸네일 이미지가 나오도록 og 이미지를 설정했다.

### 트위터 카드

트위터(X)에서 링크 공유 시 표시되는 카드다. og 태그와 별도로 동작하지만, `images`를 생략하면 og 이미지를 fallback으로 사용한다. 별도 이미지를 만들 필요 없이 og 이미지와 공유해서 쓰기로 했다.

### opengraph-image.tsx — 사이트 기본 og 이미지

Next.js App Router는 `opengraph-image.tsx` 파일을 자동으로 og 이미지로 인식한다. `ImageResponse`로 JSX를 이미지로 변환해준다.

```tsx
// app/opengraph-image.tsx
import { ImageResponse } from "next/og";

export const runtime = "edge";
export const size = { width: 1200, height: 630 };
export const contentType = "image/png";

export default function OgImage() {
  return new ImageResponse(
    <div style={{ backgroundColor: "#020617", ... }}>
      <h1 style={{ color: "#f8fafc" }}>Playground 🤾</h1>
      <p style={{ color: "#94a3b8" }}>실무에서 새로운 기능 실험까지!</p>
    </div>,
    { ...size },
  );
}
```

파일명이 정확해야 Next.js가 자동으로 인식한다. `OpengraphImage.tsx`처럼 대문자로 쓰면 일반 컴포넌트로 취급되어 동작하지 않는다.

### 포스트별 동적 og 이미지

포스트 상세 페이지에는 썸네일 유무에 따라 다른 og 이미지를 생성했다.

```tsx
// app/blog/posts/[slug]/opengraph-image.tsx
export default async function OgImage({ params }: Props) {
  const post = publishedPosts.find((p) => p.id === slug);

  // 썸네일 있으면 → 썸네일 이미지 + 제목 오버레이
  if (post?.thumbnail) {
    return new ImageResponse(
      <div>
        <img src={`${BASE_URL}${post.thumbnail}`} ... />
        <h1>{post.title}</h1>
      </div>
    );
  }

  // 썸네일 없으면 → 기본 디자인 + 제목/설명
  return new ImageResponse(
    <div style={{ backgroundColor: "#020617" }}>
      <h1>{post?.title}</h1>
      <p>{post?.description}</p>
    </div>
  );
}
```

Velite 스키마에 `thumbnail` 필드를 추가했다.

```typescript
thumbnail: s.string().optional(),
```

md 프론트매터에서 이렇게 쓴다.

```yaml
thumbnail: "/images/thumbnails/my-post.png"
```

## sitemap.xml

검색 엔진에 사이트의 페이지 목록을 알려주는 파일이다. 없으면 크롤러가 링크를 직접 따라가면서 페이지를 찾아야 해서 색인이 느리거나 누락될 수 있다.

`app/sitemap.ts` 파일을 만들면 Next.js가 자동으로 `/sitemap.xml`을 생성한다.

```typescript
// app/sitemap.ts
import type { MetadataRoute } from "next";
import { publishedPosts } from "@/entities/post/lib/posts";
import { BASE_URL } from "@/shared/lib/constants";

export default function sitemap(): MetadataRoute.Sitemap {
  const staticRoutes = [
    { url: BASE_URL, changeFrequency: "monthly", priority: 1 },
    { url: `${BASE_URL}/blog/posts`, changeFrequency: "weekly", priority: 0.8 },
    {
      url: `${BASE_URL}/blog/series`,
      changeFrequency: "weekly",
      priority: 0.8,
    },
    { url: `${BASE_URL}/hub`, changeFrequency: "weekly", priority: 0.7 },
  ];

  const postRoutes = publishedPosts.map((post) => ({
    url: `${BASE_URL}${post.url}`,
    lastModified: new Date(post.date),
    changeFrequency: "monthly",
    priority: 0.6,
  }));

  return [...staticRoutes, ...postRoutes];
}
```

`publishedPosts` 기반으로 생성하기 때문에 글을 새로 발행할 때마다 자동으로 sitemap에 포함된다.

`priority`는 페이지 중요도(0~1), `changeFrequency`는 얼마나 자주 바뀌는지 힌트다. 글은 한번 쓰면 잘 안 바뀌니까 `monthly`, 목록 페이지는 글이 추가될 때마다 바뀌니까 `weekly`로 설정했다.

## robots.txt

크롤러에게 어떤 페이지를 크롤링해도 되는지 알려주는 파일이다.

```typescript
// app/robots.ts
import type { MetadataRoute } from "next";
import { BASE_URL } from "@/shared/lib/constants";

export default function robots(): MetadataRoute.Robots {
  return {
    rules: {
      userAgent: "*",
      allow: "/",
      disallow: "/api/",
    },
    sitemap: `${BASE_URL}/sitemap.xml`,
  };
}
```

전체 허용하되 `/api/` 경로만 차단했다. `sitemap` 위치도 함께 명시해서 크롤러가 sitemap을 바로 찾을 수 있게 했다.

## JSON-LD

구글에 "이 페이지는 블로그 글이고, 작성자는 누구고, 언제 썼어"를 구조화된 데이터로 알려주는 방식이다. 잘 설정되면 검색 결과에서 리치 스니펫으로 표시될 수 있다.

포스트 상세 페이지에 `BlogPosting` 스키마를 추가했다.

```tsx
// app/blog/posts/[slug]/page.tsx
const jsonLd = {
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  headline: post.title,
  description: post.description ?? post.title,
  datePublished: post.date,
  dateModified: post.date,
  url: `${BASE_URL}${post.url}`,
  author: {
    "@type": "Person",
    name: "이지민",
    url: BASE_URL,
  },
  keywords: post.tags?.join(", "),
  ...(post.thumbnail && { image: `${BASE_URL}${post.thumbnail}` }),
};

return (
  <>
    <script
      type="application/ld+json"
      dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
    />
    <PostContent ... />
  </>
);
```

`thumbnail`은 있을 때만 포함되도록 했다. [search.google.com/test/rich-results](https://search.google.com/test/rich-results)에서 URL을 입력하면 JSON-LD가 정상적으로 인식되는지 확인할 수 있다.

## Google Analytics

방문자 데이터를 수집하는 도구다. 어떤 글이 검색에서 유입되는지, 방문자가 어디서 오는지 확인할 수 있다. SEO 작업의 결과를 측정하기 위해 함께 설정했다.

```bash
pnpm add @next/third-parties --filter web
```

```tsx
// app/layout.tsx
import { GoogleAnalytics } from "@next/third-parties/google";

export default function RootLayout({ children }) {
  return (
    <html>
      <body>...</body>
      <GoogleAnalytics gaId={process.env.NEXT_PUBLIC_GA_ID!} />
    </html>
  );
}
```

`GoogleAnalytics`는 `</html>` 바로 앞에 두는 게 맞다. `body` 안에 두면 렌더링 성능에 영향을 줄 수 있다.

```bash
# .env
NEXT_PUBLIC_GA_ID=G-XXXXXXXXXX
```

Vercel 대시보드 환경변수에도 등록해야 배포 환경에서 동작한다. 연결 확인은 GA 대시보드 **실시간** 탭에서 블로그에 접속해보면 방문자 수가 올라오는 걸로 확인할 수 있다.
