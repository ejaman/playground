# 실행 컨텍스트와 호이스팅

> 코드가 실행되면 실행 컨텍스트가 생성됨. 이때 호이스팅이 발생하여 변수들이 메모리에 등록됨.  
> `let/const`는 이 시점에 TDZ에 묶여 안정성을 확보하고, 함수는 통째로 메모리에 올라감

**1️⃣ 실행 컨텍스트**: 기록이 저장되는 물리적 공간이자 코드 실행에 필요한 모든 정보를 담은 관리 객체  
**2️⃣ 호이스팅**: 무엇을 메모리에 기록하는가

## 실행 컨텍스트

모든 자바스크립트는 코드는 실행 컨텍스트 안에서 돌아간다.  
자바스크립트 엔진은 코드가 실행되기 전, 전체 코드를 훑으며 실행 컨텍스트를 생성한다.

### 1. Lexical Environment

실제로 변수와 함수가 저장되는 곳으로 여기에는 두 개의 중요 요소가 있다.

- 환경 레코드(Environment Record)
  : 현재 컨텍스트 내부의 변수 식별자, 함수 선언 등을 기록함 ➡️ 호이스팅이 일어나는 실제 장소
- 외부 환경 참조(Outer Environment Referenve)
  : 상위 스코프(나를 감싸고 있는 바깥 환경)의 렉시컬 환경을 가리킴 ➡️ 이를 통해 스코프 체인이 형성

### 2. Variable Environment

렉시컬 환경과 거의 유사하지만, `var`로 선언된 변수만을 관리  
➡️ 현대 자바스크립트는 `var(함수 스코프)`와 `let/const(블록 스코프)`를 다르게 처리해야 함

> 왜 다르게 처리해야 할까?
> 기존 작성된 웹사이트를 훼손하지 않으면서(하위 호환성), 현대적 프로그래밍의 안전성(블록 스코프)을 도입하기 위해

### 3. This Binding

해당 컨텍스트에서 `this` 키워드가 어디를 가리킬지 결정
➡️ 전역 컨텍스트에서는 `window`를, 함수 컨텍스트에서는 함수가 어떻게 호출되었느냐에 따라 결정

### 컨텍스트의 생성과 소멸

코드가 실행되면 이 컨텍스트들은 어떻게 관리될까?

1. 코드가 처음 실행될 때 전역 실행 컨텍스트가 생성되어 콜 스택의 바닥에 쌓이고 앱이 종료될 때까지 유지된다.
2. 함수가 호출될 때마다 새로운 함수 실행 컨텍스트가 생성되어 콜 스택의 가장 위에 쌓임(JS 엔진은 항상 최상단 컨텍스트를 기준으로 코드를 실행함)
3. 함수 실행이 완료되면 해당 컨텍스트는 콜 스택에서 pop됨

> **🗺️ Call Stack**  
> JS 엔진이 "지금 어디를 실행 중인지", "이 작업이 끝나면 어디로 돌아가야 하는지" 기록하는 지도이자 작업대  
> 전역 실행 컨텍스트는 앱이 시작될 때 콜스택의 가장 바닥에 생성되고 이후 호출되는 모든 함수는 그 위에 층층이 쌓임  
> ➡️ 콜스택은 실행 컨텍스트를 관리하는 물리적 공간
>
> **Push 함수 호출 - 실행 컨텍스트 생성 ➡️ Excute 최상단 컨텍스트 실행 ➡️ Pop 실행 종료 - 해당 컨텍스트 콜스택에서 제거**
>
> 엔진은 바로 아래에 있는 컨텍스트로 돌아가 중단된 지점부터 다시 실행
>
> **⛓️‍💥 실행 컨텍스트와 싱글 스레드**
>
> - 자바스크립트는 단 하나의 콜스택을 가진다. ➡️ 한 번에 하나의 실행 컨텍스트만 처리할 수 있다.
> - 콜스택도 메모리 용량이 정해져 있다. ➡️ "Maximum call stack size exceeded"
>
> **📝 에러 로그(Stack Trace)**  
> : 콜스택이 실행 경로를 모두 기록하고 있기 때문에 가능 ➡️ 디버깅
>
> 콜스택은 동기적으로 동작한다. 따라서 만약 콜스택 상단에 시간이 아주 오래 걸리는 작업이 올라가면, 그 아래에 있는 전역 컨텍스트나 다른 함수들은 실행되지 못하고 멈춤 ➡️ 블로킹(Blocking) 현상  
> ➡️ 이벤트 루프와 태스크 큐 활용

## 호이스팅

호이스팅을 이해하기 위해선 환경 레코드(Environment Record)에 변수 식별자를 등록하는 과정을 살펴봐야 한다.

### var

- 엔진은 `var`키워드로 선언된 변수를 찾아 메모리 공간을 확보하고, 해당 식별자를 환경 레코드에 기록한다 ➡️ 호이스팅
- `var` 변수의 경우, 이 생성 단계에서 식별자 등록과 동시에 `undefined`로 초기화를 한다.  
   ➡️ 따라서 물리적 메모리 공간에는 이미 `undefined`라는 값이 기록되어 있는 상태

<br/>

**🤾 호이스팅의 실체**

선언문이 코드 상단으로 물리적으로 이동하는 것이 아니라 실행 전 이미 메모리 준비가 끝난 것

> `console.log(variable);`
>
> 1. JS 엔진은 현재의 실행 컨텍스트 내 환경 레코드에서 해당 식별자를 찾음
> 2. 이미 생성 단계에서 식별자가 등록 + `undefined`로 초기화 ➡️ 유효한 변수로 판단
> 3. 참조 에러를 던지는 대신, 메모리에 기록된 `undefined`를 반환
> 4. `var variable = 1;`에 도달하면 `undefined`였던 메모리 값을 실제 값으로 업데이트

### let, const

`var`와 마찬가지로 `let`, `const`도 호이스팅 된다.

- 실행 컨텍스트의 생성 단계에서 환경 레코드에 식별자를 등록한다. ➡️ 엔진이 이 변수의 존재를 인지함
- 생성 단계에서 메모리 공간은 확보될 수 있으나 초기화되지 않는다.

> `console.log(variable);`
>
> 1. JS 엔진은 현재의 실행 컨텍스트 내 환경 레코드에서 해당 식별자를 찾음
> 2. 이미 생성 단계에서 식별자가 등록 but 초기화 ❌
> 3. JS 엔진에서 초기화되지 않은 메모리에 접근하는 것은 금지 ➡️ `ReferenceError` 발생
> 4. `const variable = 1;`에 도달하면 비로소 초기화가 이루어지며 변수 사용 가능

> **일시적 사각지대 TDZ**  
> JS에서 변수가 선언되었지만, 초기화는 되지 않아 접근할 수 없는 구간  
> ➡️ 스코프 시작 지점부터 변수 선언문이 나오기 직전까지의 구간
> ➡️ 이 때 변수에 접근하면 JS 엔진이 `ReferenceError` 던짐
