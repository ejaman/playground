# 스코프 체인과 클로저

> 함수가 호출될 때마다 새로운 컨텍스트가 생김. 이때 "내 주변에 변수가 없으면 나를 감싼 외부로 가서 찾자"는 스코프 체인이 형성됨. 만약 내부 함수가 살아있는 동안 외부 변수를 계속 잡고 있다면, 그것이 바로 클로저가 되어 캡슐화를 실현

**1️⃣ 스코프 체인** : 식별자를 검색하기 위해 현재 렉시컬 환경에서 상위 스코프로 거슬러 올라가는 탐색 경로  
**2️⃣ 클로저** : 외부 함수가 종료되어 콜스택에서 사라져도, 내부 함수가 그 환경을 기억하고 있는 상태

<br/>

스코프 체인과 클로저를 알아보기 전 먼저 렉시컬 스코프가 뭔지 알아야 한다.

## 규칙 - 렉시컬 스코프(Lexical Scoping)

JS 엔진이 식별자를 찾는 범위를 결정하는 근본적 규칙으로  
함수를 어디서 호출했는지가 아니라, **어디서 선언**했는지에 따라 상위 스코프가 결정된다.  
➡️ 개발자가 코드를 타이핑하는 그 물리적 위치가 스코프를 결정함

> ⬇️ 코드를 실행하기 전, 컴파일 단계에서 함수가 어디에 적혀 있는지 확인한다.  
> ⬇️ 함수 선언문을 발견하면 해당 함수가 속한 "물리적 환경"을 파악한다.  
> ⬇️ 해당 함수의 실행 컨텍스트가 생성될 때, 외부 환경 참조에 부모 스코프 주소를 미리 저장한다.

코드를 작성하는 시점에 상위 스코프가 확정되므로, 개발자가 소스 코드를 눈으로 봤을 때 변수가 어디서 오는지 예측 가능하게 만든다.  
➡️ 호출 위치와 상관없이, 선언된 위치의 부모를 찾아가는 일관된 탐색 경로를 보장할 수 있다.  
(스코프 체인이 만들어질 때 어디로 연결될지를 결정하는 설계도 역할)

## 실행 - 스코프 체인(Scope Chain)

현재 실행 컨텍스트의 렉시컬 환경에서 변수를 찾지 못하는 경우, 외부 환경 참조(Outer Environment Reference)를 따라 상위 스코프로 타고 올라가며 변수를 검색하는 물리적 탐색 경로이다.  
➡️ 렉시컬 스코프 규칙에 따라 미리 정해진 상위 환경 주소를 따라 갈 수 있는 것!

JS는 어떻게 식별자를 결정하는걸까?  
JS 엔진은 코드 안에서 변수나 함수 이름을 만났을 때 이 이름이 가리키는 실제 메모리 주소가 어디인지를 찾아 식별자를 결정한다.

> **console.log(name);**
>
> 현재 실행 중인 실행 컨텍스트의 환경 레코드를 뒤진다.  
> ⬇️ 여기 name이 있는 경우 🙆‍♀️ - 해당 식별자가 가리키는 값을 즉시 반환 ➡️ 결정 완료  
> ⬇️ 여기 name이 없는 경우 🙅 - 외부 환경 참조를 타고 상위 스코프로 이동
>
> 전역에 도달할 때까지 이 과정 반복  
> 끝까지 못 찾으면 🤾 `ReferenceError`

## Result! - 클로저(Closure)

외부 함수보다 내부 함수가 더 오래 유지되는 경우, 외부 함수의 지역 변수를 여전히 참조할 수 있는 상태를 말한다.

> **참조(Reference)**
>
> 함수가 실행을 마치면 실행 컨텍스트는 콜스택에서 제거된다.  
> 그러나 실행 컨텍스트가 사라진다고 그 안에 렉시컬 환경이 바로 제거되지는 않는다.
>
> 😊 JS의 가비지 컬렉터는 어디선가 참조되고 있는 메모리는 해제하지 않는다.  
> 따라서 내부 함수가 외부 함수의 변수를 참조하고 있다면 외부 함수가 종료되어도 그 렉시컬 환경은 메모리에 남는 것!

렉시컬 스코프 덕분에 외부에서 내부 변수를 건드릴 수 없고, 클로저 덕분에 내부 상태를 안전하게 유지할 수 있습다.

### 🚨 주의

더 이상 필요하지 않은 클로저가 상위 스코프를 잡고 있다면 메모리 낭비가 발생한다.  
따라서 클로저를 참조하는 변수에 `null`을 할당해 참조를 끊어주면, 가비지 컬렉터가 해당 메모리를 수거할 수 있다.
